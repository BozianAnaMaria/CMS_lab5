# Raport: Lucrarea de laborator nr. 5. Securitatea WordPress

## Scopul lucrării
Consolidarea celor mai importante practici de securitate în WordPress prin gestionarea rolurilor, aplicarea tehnicilor de hardening, monitorizarea activității, și configurarea avansată folosind pluginul All In One WP Security & Firewall (AIOS) în combinație cu instrumente dedicate pentru backup (UpdraftPlus). Acest demers are ca obiectiv final transformarea unei instalări WordPress clasice (potențial vulnerabile) într-un sistem robust, monitorizat, apărat la diferite straturi de atac și protejat împotriva pierderilor de date, fiind complet pregătit pentru mediul de producție.

### Pasul 1. Pregătirea mediului de lucru

Acțiunea inițială a fost configurarea corectă a mediului de dezvoltare. Am accesat instalarea locală de WordPress și m-am autentificat cu contul de administrator principal. 
Pentru a permite sistemului afișarea eventualelor erori ascunse sau avertismente critice pe durata testelor detaliate ale platformei (extrem de util pentru un firewall extins ca AIOS care poate bloca funcții neintenționat), am activat modul de depanare centralizat. Am navigat la locația instalării de pe hardisk, am deschis fișierul `wp-config.php` din rădăcina site-ului folosind un editor de text local și am asigurat configurarea prezenței următoarei directive (la care am asigurat valoarea true):


Această setare ne asigură vizibilitate maximă în cazul apariției conflictelor de securitate.

### Pasul 2. Gestionarea rolurilor și politicilor de parole stricte

Limitarea privilegiilor este o primă acțiune defensivă, realizată conform principiului „Principle of Least Privilege”. Aceasta izolează atacurile post-explotare, limitând gravitatea efectelor de tip defacement sau obținerii RCE (Remote Code Execution).
Am navigat în panoul de administrare la secțiunea **Users (Utilizatori) -> Add New (Adaugă utilizator nou)**. Cu scopul de a izola activitățile cu impact general pe site și a nu le comasa exclusiv sub puterea super-admin-ului, am generat un utilizator adițional limitat, dotat cu rolul de **Autor**. Acest cont este dedicat publicării unice de conținut curat și nu va avea nicio interfață privind adăugarea setărilor nucleare ori instalarea plugin-urilor.  

![Interfața de configurare și de adăugare a noului cont de utilizator cu nivel de Autor](images/image.png)

Confirmarea completă a înregistrării s-a materializat odată ce acest utlizator Autor a fost generat și a apărut prezentat distinct în lista unificată de administrare. Aici putem vizualiza direct, într-o manieră integrată, coexistența administratorului alături de utilizatorul cu privilegii reduse.
Am constatat și asigurat faptul că parola pre-asignată utilizatorului creat este extrem de complexă, ambele instanțe preluând motorul propriu WordPress pentru generarea de parole impuse automat de grad "Strong" – codificate cu o pondere aleatorie semnificativă de litere mari, litere mici, cifre greu prezicte și simboluri asortate care depășesc lungimea riscantă de 8 caractere tipice.

Cea mai des utilizată poartă de intrare a atacatorilor botnet este identificarea și exploatarea unei versiuni de element CMS învechite (cunoscută ca CVE).
Accesând meniul structurat sub elementul **Dashboard -> Updates (Panou de control -> Actualizări)** m-am asigurat, direct din sursele oficiale, că instalarea curentă utilizează în mod activ ultima versiune stabilă a pachetului WordPress Core. Twenty Twenty-Four, sau teme clasice incluse automat) și eventualele componente plugin erau strict „Up to date”.

### Pasul 4. Hardening de bază (Restricționarea editării și limitarea accesului web server)

Una dintre vulnerabilitățile frecvente și periculoase ca abordare Post-Exploitation este capacitatea unui Administrator (ori mai critic a unui cont superior preluat malițios) să modifice și să injecteze direct cod de la distanță (shells/backdoors care pot rula via PHP) pur și simplu intrând natural în editorul de teme asimilat nativ prin Appearance -> Theme Editor din dashboard-ul general. Pentru a sigila acest proces extrem, am revizitat fișierul cheie `wp-config.php` și am impus directiva necesară ca să eliminăm complet componenta vizuală de editare internă accesibilă din interfață, securizând tot codul. 

```php
define('DISALLOW_FILE_EDIT', true);
```

Mergând un nivel mai adânc de protecție pură la nivelul protocolului de server web HTTP (folosind sub-sistemul Apache), am limitat sever și imediat expunerea și potențialul de acces web HTTP extern referitor spre locația la care rezidă `wp-config.php`. Fără impunerea acestei proceduri restrictive sub ierarhia fișierului web explicit `.htaccess`, variabilele setărilor delicate ale bazei de date (DB Host, Use, Password) și tokenurile dinamice generate de salt-urile criptografice arhivate acolo, ar prezenta riscul extrem să fie compromise, interogate de hackeri prin alte instanțe nesecurizate care ar staționa local hostate izolat sub server. Prin injectarea blocului restrictiv pe nivel sistem fișiere este acoperit orice bypass de permisiune:

```apache
<files wp-config.php>
   order allow,deny
   deny from all
</files>
```

Această dublare asigură că indiferent dacă este identificată platforma, ele nu pot elucida credentialele centrale de legătură la MySQL, acțiune care neutralizează direct intruziunea critică bazată fișiere brute de sistem. Opțiunea din dashboard vizual „Theme editor” care ar compune editările se evaporă efectiv sub re-încărcarea panoului.

![Codul definitivelor de restrictionare a accesului în fișierul Apache .htaccess sub blocul de siguranță asigurat pentru modulul fișierelor](images/image%20copy%204.png)


### Pasul 5. Instalarea și configurarea avansată folosind modulul All In One WP Security & Firewall (AIOS)

Peste blocarea manuală primară am considerat critic necesar extinderea măsurilor superioare care necesită monitorizare vizuală complexă bazată pe detecție automată, operare asimilată de instrumente centralizate de audit WAF (Web Application Firewall). Drept urmare, a fost instalată și declarată activă platforma superioară **All In One WP Security & Firewall (AIOS)**. Interfața inițială integrată aduce undă verde ce confirmă procedeele fundamentale ce puseseră stăpânire de acțiune protejând mediul pre-operațional în fundal.

![Panoul finalizat de întâmpinare cu mesaj de succes ulterior primei proceduri de activare a măsurilor obligatorii preinstalate de AIOS (You're all set)](images/image%20copy%205.png)

Pentru o rată solidă de acoperire care să depășească gradul mediocru de atac, am rearanjat și manipulat o listă stufoasă de opțiuni din configurația plugin-ului, conform practicilor următoare:

#### A. Manipularea conturilor de utilizatori vulnerabile (Renamed Admin - module User Accounts)
În mod repetat s-a demonstrat la scară largă cum roboți de botnet adunați din mii de device-uri asediază constant punctele de intrare testând variante standard printre care „admin” care este predominant știut ca referit ca numele inițial de profil. 
Navigând la panelul **User Accounts**, sub atenționarea masivă marcată specific de AIOS în chenar roșu, s-a efectuat direct comuntarea numelui administratorului de uz curent din acel moment „admin” cu o denumirea randomizată care preia identitatea `admin132`.
Operațiunea simplă produce efectiv invizibilitate reală asupra celui mai proeminent user standard contra celor mai mari arhive și dicționare dedicate atacului via bruteforcing de utilizatori cunoscute în atacul asupra structurilor CMS default.
![Instanța nativă AIOS recomandă imperios și execută modificarea schimbării contului super-admin asimilat din oficiu cu referință simplă](images/image%20copy%208.png)

#### B. Restricția procesului de înregistrare automată (Manual Approval Registration - Spam Control)
Ca element preventiv când s-ar putea activa permisiunile vizitatorilor de a realiza proprii utilizatori în mediu. Prin blocarea înregistrărilor se vor evada spammerii umani dedicați prezenței cu scop de injecții cu mesaje nedorite care solicită link-uri referral ca promovare blackHat SEO. Setarea a devenit restrictivă cu trecerea la acceptare de flux sub opțiunea **Manual Approval**. Acest concept generează trecerea direct în baza limitată de așteptări de confirmare și reține userii spam exclusiv rezervați listei ca stare "Pending". Neavând nici-o verificare umană pozitivă apărută pe email care confirmă trecerea ei rămân blocați definitiv fară log-in și postare pe WordPress.
![Sistemul ce dovedește detaliat configurarea manualizării cu rol esențial ca filtru la primirea noilor membrii exclusiv](images/image%20copy%209.png)

#### C. Politici anti-Brute-Force și rate limitare IP (Login Lockdown Module)
Mecanismele moderne care monitorizează intruziunile opresc scripturile și crawlerii rapizi prin impuneri severe limitative de repetiție a conectării per secundă de trafic. Sub ramura **Login Lockout**, am adoptat impunerea și blocajul terminal definit de: opțiune max eșec de **5** treceri de eșec false; dacă botnetul insistă sub marja ferestrei stricte de contorizare per valoare **15 minute**, motorul rate-limit impune prin IP tăierea integrală (lockout) fără excepții timp general de **30 minute**, iar accesatorii primi repetitivi devin blocați chiar pe un prag maxim total de 60 de minute.
![O previziune referentă direct setărilor aplicate sever ale ferestrei interval de retry scurte ca permisiuni pe timpi de respingere asigurați de lockdown (30m)](images/image%20copy%206.png)

#### D. Extragerea sesiunilor paralele latente expuse prin deconectare automată (Force Logout Control)
Un nivel complex care combate furtul și replicarea per sesiune HTTP nedetectate acționează la expirarea obligatorie. Într-o situație obișnuită lăsăm fără vizualizare panoul intern care prin Cookie HTTP ar conține încă sesiunea conectată, deschisă oricui manipulează sistemul gazdă sau stația vizuală web. Așadar protecția se ridică cu modulul reglat pe **Force user logout options** introducând deconectarea generalizată pe absolut toata platforma neaccesorizată indiferent de permisiuni. Această valabilitate e redusă ca resetare periodică generală direct sub 1440 minute reprezentând re-logare absolut totală ce ocolește și invalidează toți tokenii expirați în interval de la depășirea duratei prag (o zi calendaristică întreagă: 24h). 
![Restricționarea sesiunilor valabile ale administrării pe prag setat limitativ periodic ce produce Forcing validare via timeout general după durata (1440m)](images/image%20copy%207.png)

#### E. Schimbarea topologiei endpoint-urilor (Mascarada via Rename Login Page Module)
O soluție foarte elegantă, eficientă de stopări ce acopere asalturi robotice ce bombardează neîncetat direct fișierul hardcodat la scara universală: `wp-login.php`. Mascarea s-a prelucrat cu componenta specializată AIOS accesată la secțiunea „Brute Force". Efectuând operațiunea simplă **"Enable Rename Login Page"**, s-a instigând interfața CMS spre direcția ascunderii panoului natural.
Pentru locația personalizată a url-ului a rezultat alegerea random sub ramura customizabilă `/login-securizat`. Noua procedură a schimbat în proporții considerabil structura URL, protejând platforma contra zecilor de atacatori simpli. Acest pas esențial mărește direct necesitatea atacatorului de a descoperi și accesa efectiv locul unde se trimite request-ul de încercare parolă. URL-ul unic este gestionat izolat local prin notițe tip Pass Manager uman.
![Setările modulelor Brute force ce certifică redirecționarea interfeței convenționale prin noul element invizibil wp (login-securizat)](images/image%20copy%2014.png)

#### F. Inspecțiile strat-bază POSIX ce acoperă nivelele OS fizice instalate (Filesystem Structure / Permissions)
Configurația depășește frameworkurile aplicațiilor per nivel abstract și accesează inspecții sub sistemele de operare instalate local. AIOS aduce un tool dedicat File Permissions capabil ca la nivel kernel să depisteze modurile și excluderea ex. de a depista ca se rula permisiunea 777 nesecurizată general. Modul are validare să insiste ca mapări să fie standard POSIX de preferat minim mod restrâns doar de execuție limitat la 755 dosare ce prezinți 644 ca element citire pe fișier normal terminal). 
Sistemul analiză ruleaza platforma prezentată confirmând ca modulul se identifică prin validare a unui *Windows server* local unde structura CHMOD nu e valabilă exact la concept Unix, prezența testelor arătând adaptabilitatea sistematică AIOS.
![Ecran de identificare sistem și de adaptabilitate ce analiază permisiunile redate specific cu modul Windows ca scanner in modul File Permission Scan](images/image%20copy%2010.png)

#### G. Integrarea ierarhiilor prin Firewall Web App și Politici HTTP speciale
Securitatea exponențială pe modul de preluare a pachetelor a acoperit o colecție specifică avansate aduse ramurii Firewall în AIOS. La prima etapă s-a adoptat generarea de politici în rândul de validare general pe infrastructură regăsit în panoul nativ Apache via injecția de **`.htaccess firewall rules`**, pe langa protectie s-a adus o constrângere de *Max file upload size* (val limitative a consumului rețelelor: redus 100mb) util prelucrarii prevenția resurselor și protecției DDoS direct în panou de filtrare.
![Ecran integrarea regulilor de limitări direct in fișier .htaccess referențiate prin submodul generic asimilat Firewall-ului](images/image%20copy%2011.png)

O detaliere mai pronunțată care lucrează cu un parser aplicativ specific pentru limbajul sursă este declarată prin intermediul zonei avansate implementate sub panoul **PHP Firewall Settings**. O politică esențial aplicată reține blocajul sub opțiunea declarativă *„Deny bad query strings”*. Această decizie devine critică apărând instanța de cereri tip HTTP pe care o recunoaște imediat ca injectări via linkuri cu argumente alterate GET/POST (conținând string-uri vizibile pe adresa link orientate exploatărilor specifice la scurgerea injectiei RCE/SQL și a pericolele exploatăroiri pe Cross-Site-Scripting).
![Setările împotriva preluării string-urilor de manipulari de input interzise referitoare la detecția XSS din tab-ul bad query parameter](images/image%20copy%2012.png)

Asigurând elementele mai nuanțate din aceeași stivă, sub **String filtering** a fost impus să se declanșează obligativitatea regulii **"Enable advanced character string filter"**. Extinderea funcționează ca senzor special care anulează, ca o politică aspră specifică, semnătura și ansamblul secvențial a seturilor pe litere asimilitate metodelor complicate de evadări specifice pattern-urilor asimilate la testele malițioase (XSS Strings Patterns inserate des prin manipulare pe mesaje formulare).
![Procedurii interne speciale String Filtering setată ca ariergarda detecțiilor secvențelor de caractere pe atacatori complecși payload XSS](images/image%20copy%2013.png)


#### H. Auditări avansate pentru schimbările coruptibile invizibile direct (Scanner Malware) & Alertele Email 
Platforma nu a omis uneltele automatizate asincrone a căror procedură necesită raportarea integrității prin monitorizare. A fost declanșat prin File security / *Files change detection settings*, procesorul de cron-scanning programat ce evaluează constant starea inițială generală. Măsura compară hashurile a ce fișiere erau anterenior stocate și a fost instigat să ruleze scanul integrat și să acționeze sistematic evaluarea o dată la "4 săptămâni". În acest fel aflăm din timp la coruperea unui fișier sub o atacator masivă fără prezența alterăriilor vizibilă a formei web (Backdoor de server rootkits stealthy).  
![Setarea de rutine Scanner cron control periodice ca proces periodic pe monitorizare la detecții e fișierelor interne editae malițioase invizivile general](images/image%20copy%2015.png)

Aproape final a adus notificarea promptă vizuală – nu era permis lăsarea incidentului de izolare netrasmis a elementului de urgență. Reglementarea **Notifications alert settings (Notify by email)** a obligat serverul la o alertare direct pe in-box-ul cu o sursă pre-asigurată dedicat profilului admnistrator (așa cum am indicat via formular e-mail) a incidentelor care dovedesc cum și cine este atacat exact când cineva a atins Lockout rate max fail attempts pe firewallul de Brute Force per sesiune din dashboard loging limitări proceduri.
![Prezența doveditelor completărilor de trimitere informației IP blocate orientativ a rapoartelor prin email alert module preconfigurat adminilor aditionali](images/image%20copy%2019.png)


### Pasul 6. Sistem Garantat Defensiv folosind Extensii Backup periodice (UpdraftPlus Module Settings)
Chiar și cu aceste ziduri imense implementate, regulamentul din securitate reiese că este strict imperios crearea unei plase a redundanței ca metodă pe date, ce va asigura refacerea site-ului de baze pierdute de accidente locale irrecuperabile a corupțiilor masive imprevizibile sau deciziile extreme ca server Wipe-up Ransomware/Wiper attack general post-compromise. Aceasta a condus prezență și pe integrarea super avansat unui set complet integrat via aplicație superioară completă cum este recunoscura platformă UpdraftPlus ca mecanismen ce generează copiere la sursă independent asigurate la retenție schemei bazei.

O primă etapă s-a orientat pe nivel planificare. Tabloul panoului la secțiunea internă a configurării ca stas a dezvoltat prin sistem o metodă automată și logică. A rezultat impunea unei **politici hibride și temporală**: unde copierea documentelor Media (Photos, Video Files backup) preiau ritmul manual dar criticul **Database backup schedule (Logurile de Baza de Date conținuturi SQL tabele core-db)** revin a rula nativ complet și izolat direct programate sub incidența activată cron (lunar referință Monthly având impunere pe maxim retenție 2 backup seturi stocabile arhivă locale prevăzute suprascriere depozitat pe spatiul rezonat preluari resurse server limit).
![Formularele Settings general administrative din planul setărilor Schedule implementat pentru Database periodice și limitative arhivari de reținere Fișiere Manual pe asistentul de proces Backup Schedule Updraft](images/image%20copy%2018.png)

Făcând trecerea cu starea operațională pe log-uri generale asigurate de prezența tab-ului (Backup/Restore tab panel), au început o executare inițială instantanee generală predefinit job pe site pregătind backup-ul. Cu proces finalizat conform sistemelor și stocate direct izolate pe referințele locale sub denumirile bazate temporal referențial. Acestea adunasera cu acuratețe arhivate pachetele și devin prelucrate și indexate complet separat ca depozit (Database archive zip separat alături de log structura componenta plugin themes must-use) menținând așezarea lor în mod integrativ clar separat cu posibilitatea a manipula doar individual de o arhivă stocată curata.
![Lista existentului tablou de stoc local generalist și arhival reținut din istoricul indexat per pacheții Backup SQL curat preluati integral local](images/image%20copy%2016.png)

Rapoartele finale au ridicat asistența completă prin manevrarea arhivelor logice DB selectate individual cu suport complet descărcabile local pentru extragerea acestora prin funcția *Download to your computer*. Componentă care oferă acuratețe garanției unei rețete offline salvate izolat ca transfer arhive date direct referențiate zip. Gata garantat direct a o prelua la nevoie Disaster recovery spre transfer curățat (Offsite-cloud remote-system host backup), o decizie care blindează structurile a nu a sta centralizate strict pe discul ce funcționează instanța curentă. S-a realizat extragerea arhivei pure g-zip offline.
![Sistemul confirmă ca fișierul bază deține stocare curățată independentă pe descărcare spre stadiul GZ fișiere de suport arhivări separate client terminalului extern offsite procedurii recovery via Updraft modal function extragere Download](images/image%20copy%2017.png)



### Pasul 7. Etapa Supremă de Certificări prin scenarii de validare (Test Blocking Brute-Force limit și Verificarea Datelor Restabile via Restore procedure)

Dovada existenței sistemulilor asigurate necesită certitudini obținute exclusiv testării negative (simularea ca penetration testing din unghiul vizat adversar negativ atacator) pe ambele brațe asimilate ca element de limită activ și validare garanție funcțională la moment post-avarie irrecuperabile date umane gresite accidental ștergere. 

**Exercițiul A. Test de izolare restrictivă prin agresiune IP blocat (Rate-Limiting firewall trigger):**
O dovadă perfectă pentru a verifica cum bariera Brute Force blindează din moment. Pentru proces s-a demis completarea log-uri (Logout integral direct web sistem profile browser izolat administrator root panel view) spre a obține vizitator guest standard simplu pe sesiune izolată din terminal web non-permisiv. Mai încolo, invocând ruta maschei specifice predefinit endpoint via adresa specializata web (`localhost/wp/login-securizat`), am manipulat atacul artificial asupra platformei, trimițănd către cont utilizator testului specificat Autor încercări intenționate brute de reîncercări negative pe câmpurile parole.  S-a generat asaltul, cu inserții text refuzate de exact sistem.
Ca răspuns imediat și de o rezonare mecanică instantă fară prelungiri de verificare o dată cu trecerea asalturilor negative permise repetate 6-6x trecute neconform parole greșite complet pe timp redus, limitările scutului pluginu-lui nativ Rate-Limiting din procedură AIOS din cloud panel a răspuns la impunere lovind cu impunerea general blocaj per sesiune blocandu-l și suprimând sursa în ecran alb curat doar pe baza indicatorului scurt panoul avertizare vizibil izolant interfeței browser referențiat clar la limitatorul specific asimilat AIOS IP blockage rules „ERROR: Access from your IP address has been blocked for security reasons...". Fără ezitare refuzul ca agresiunea să poată intra vreo rută de funcția HTTP core API pe sistem login era irefutabil implementată pe un proxy firewall complet și solid ignorând accesul complet total.  
![Vizualizarea efectelor confirmante la excluderea direct din platforma panoului asigurat izolat la interceptie repetari asalt refuz cu limitator text alertă general simplificat](images/image%20copy%2023.png) 

Deoarece un analizator trebuie să studieze incidentele și cum sa captat referințele în sistem firewall pentru raportare, folosind ca terminal administrator separat mediu (o deschidere conectată izolat cu permisiunile menținute ca profile pre-verificate IP) a redat în consola WP panoul extins al platformei AIOS log activități incident monitor meniu denumit „Dashboard sub *Locked IP addresses*”. 
Interfața a arătat extrem de rapid depistarea exact precis cu precizie a originii protocol surselor adresa IP asimilate (atacantului originar simulat browser paralel) izolând complet pe tabelă de avertismente o listarea complet blocată referitoare ca justificate (Reason logs indicator tab text: *"too_many_failed_logins"*), plus timpi reîntoarcere restabilită de pe cron. Acest scenariu testează perfect cum auditările stochează eficient loguri preluând integral și sigur informațiile blocat. 
![Meniurile demonstrează centralizatorul rapoartelor procedurale la închiderea agresiunilor stivuind tabele referențiale interne și dovedind eficienta izolare alocat pe tablou activ lockouts per surse adrese](images/image%20copy%2024.png)


**Exercițiul B. Procedură extremă de recuperare și readucerea consistenței stării (Avarii provocate intern / Ștergere a articolelor referințele Post Content Destroy Test via Undo Backup Recovery):**  
După protejarea sistemelor din afară o cauză comună pe breșe sau pierderilor de bază stă manipularea și dezastrul incidenturilor erori greșite umane procedurale distrugeri conținut nedorite. A condus planuri evaluării prin deces/înlăturarea reală asigurate pe tabloul instanței articole a pagini conținut web.

## Partea a II-a: Răspunsuri la întrebările de control

**1. De ce DISALLOW_FILE_EDIT și permisiunile corecte pe wp-config.php reduc semnificativ riscul post-exploit?**  
Dacă un atacator obține acces la contul de Administrator (prin phishing de sesiuni compromise, scaneri sau brute-force de proporții direcționat), prima lui acțiune este de obicei rapidă - caută o cale la injectarea de cod malițios rullabil invizibil (backdoors specifice pe stivă shell pe execuției). Proces devine banal instalând prin modificatorul integrat pre-setat ca editor de cod aferent direct panoului administrare WordPress (fără a mai trebui pe host pe alte conturi OS sa fenteze panouri). Astfel codificator PHP devin valabile pe pluginuri/machete temă cu permisiunea sistem de scrieri de la nivel server la click distanță. Setarea valorii simple cu constantă adevărată prin cod la directivă `DISALLOW_FILE_EDIT` elimină radical posibilițatea asta ca acel meniu complet din aplicație editor frontend dispare de interfață grafică și restrictionează procesările pe modificări edit script pe nivel root, blocând masiv prima tentativă exploataților pe backdoor de interfață control. În rezonanță cu asta ca pe protecția chei nucleice, izolarea referită ca indexul configurativ pe permisiunile extrem stricte la documentul de configurare centrală setări private DB config OS locale pe sursă baza platformei redate la document document referent pe sistem root document server `wp-config.php` (cum a fi setarea procedur indexarea generalizata modul specific index ca POSIX reguli drept la val 644 ca write/read si 440 la mod read extrem restanger restriction strict pe utilizatori grupe web grup proces), ce izolează suplimentar din OS protecțiile referind, alături masurilor protecțiilor re-alocări setare HTTP redirect rules restriction HTTP pe server Apache nativ la bloc din `.htaccess` previziunare serverului, toate asimilând interdicții totale și respingătoare în așa manieră că se împiedică simplii utilizatorii izolați de a descarcă pe URL sau prelua execuția din alte scripturi paralele exploate ce stau vulnerabile alături cont pe hostname c-panel root sau alte CMS server adiacente site de pe host share environment, ce ar blinda hackerilor citirea furturi credentialele databaze parole în clare logurilor necriptate sau extragere tokenurile cheile de protecție API criptare salts instalate vulnerabili ascunse pe configurațiile locale de conținut ascuns fișiere baza stâlp config la root locație locale.

**2. Ce setări ai ales pentru Login Lockdown/Firewall și de ce (explică echilibrul între securitate și experiența utilizatorului)?**  
Configurația aleasă și motivată strategic de limitatoare prag pe modul specific Rate-Limiting ca formă general e adoptarea log-setărilor de referință cu un număr moderat maximum restrâns la calculat un set de repetare fix de doar 5 încercări premise și lăsate executate false, grupate ca procedură reîncercare cu contor reținut pe un fereastră strict operațional memorie reținut memoriei alocației fix 15 minute limitative fereastra de erori IP sesiune bloc.
